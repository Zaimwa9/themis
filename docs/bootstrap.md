# Automated setup

`python -m themis init` uses GitHub's App Manifest flow. It creates a private
GitHub App with Themis's exact permissions and events, exchanges GitHub's
one-time code for the App credentials, verifies installation on the requested
repository, and writes `.env`, `compose.yaml`, and the model credential volume.

GitHub still displays two authorization screens: one to create the App and one
to choose repositories. Those ownership decisions cannot be made through the
GitHub CLI or REST API. Everything before, between, and after them is handled
by the bootstrap.

## Public deployment

Run this on the machine that will host Themis:

```bash
mkdir themis-deploy
docker run --rm -it \
  --user "$(id -u):$(id -g)" \
  -p 127.0.0.1:8976:8976 \
  -v "$PWD/themis-deploy:/output" \
  -v "$HOME/.codex:/host-codex:ro" \
  ghcr.io/zaimwa9/themis:latest \
  python -m themis init \
  --repo OWNER/REPO \
  --organization OWNER \
  --public-url https://themis.example.com \
  --output /output \
  --codex-auth /host-codex/auth.json \
  --bind-host 0.0.0.0 \
  --no-browser
```

Omit `--organization OWNER` for a repository owned by your personal account.
The callback container listens on all container interfaces, but Docker publishes
it only on the host loopback address. Keep that `127.0.0.1` restriction.

The command prints `http://127.0.0.1:8976/`. Open it in the browser where you
are signed in to GitHub. Confirm the generated App name, then select the exact
repository supplied to `--repo`. The bootstrap independently checks that the
installation callback belongs to that repository before completing.

The success page and terminal both show the generated bot mention, for example
`@themis-acme-123`, and the command used to request a review. The same identity
is saved in the generated `themis-info.json`, so users and automation can find
it again without reading logs or credentials.

Start the generated deployment:

```bash
cd themis-deploy
docker compose up -d
```

## Local tunnel

An ngrok account and auth token are required for the bundled tunnel:

```bash
export NGROK_AUTHTOKEN=<your-token>
mkdir themis-deploy
docker run --rm -it \
  --user "$(id -u):$(id -g)" \
  -e NGROK_AUTHTOKEN \
  -p 127.0.0.1:8976:8976 \
  -v "$PWD/themis-deploy:/output" \
  -v "$HOME/.codex:/host-codex:ro" \
  ghcr.io/zaimwa9/themis:latest \
  python -m themis init \
  --repo OWNER/REPO \
  --organization OWNER \
  --tunnel \
  --output /output \
  --codex-auth /host-codex/auth.json \
  --bind-host 0.0.0.0 \
  --no-browser
```

Then start the tunnel profile:

```bash
cd themis-deploy
docker compose --profile tunnel up -d
```

The manifest starts with a placeholder webhook URL. Once ngrok is ready,
Themis discovers its HTTPS URL and updates the App webhook. The webhook secret
generated by GitHub is preserved.

## Claude engine

Pass `--engine claude` and omit `--codex-auth`. After running `claude
setup-token`, place the token in the generated `.env`:

```text
CLAUDE_CODE_OAUTH_TOKEN=<token>
```

The generated Compose file exposes it only to the isolated agent service.

## Files and recovery

The bootstrap writes:

- `.env`, mode `0600`, containing the App credentials and generated internal
  agent token.
- `compose.yaml`, using the published Themis image rather than a source build.
- `themis-info.json`, containing the non-secret App slug, `@mention`, and target
  repository.
- `codex-seed/auth.json`, mode `0600`, when Codex authentication was found or
  supplied. A one-time Compose init service copies it into the persistent
  `codex-home` volume without exposing it to the controller.

It refuses to overwrite an existing `.env` or `compose.yaml`. If App creation
succeeds but installation is cancelled, the credentials remain in `.env` and
the App remains in GitHub. Install that App on the requested repository and
start the deployment, or delete the App and the incomplete output directory
before restarting bootstrap.

## Running from a checkout

For development, the same flow can be run without the bootstrap container:

```bash
uv run python -m themis init \
  --repo OWNER/REPO \
  --organization OWNER \
  --public-url https://themis.example.com \
  --output ./themis-deploy
```

The local command opens the browser automatically unless `--no-browser` is
passed. `--callback-host`, `--callback-port`, and `--bind-host` are available
for unusual container or remote-host networking arrangements.

## Manual fallback

To create the App manually, configure:

| Setting | Value |
|---|---|
| Webhook URL | `https://HOST/webhook` |
| Contents permission | Read-only |
| Pull requests permission | Read and write |
| Issues permission | Read and write |
| Events | `pull_request`, `issue_comment`, `pull_request_review_comment` |

Generate a private key, install the App on the target repositories, and set
`THEMIS_GH_APP_CLIENT_ID`, `THEMIS_GH_APP_PRIVATE_KEY` (PEM or base64),
`THEMIS_GH_WEBHOOK_SECRET`, and a random `THEMIS_AGENT_TOKEN` as described in
[`configuration.md`](configuration.md).
